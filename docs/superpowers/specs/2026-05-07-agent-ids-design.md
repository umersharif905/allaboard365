# Agent IDs — Design

**Date:** 2026-05-07
**Branch:** `feat/agent-ids`
**Status:** Draft, awaiting user review

## Summary

Introduce a tenant-scoped, human-readable, sequentially-generated `AgentCode` for every agent in the system, mirroring the existing Member ID pattern (`oe.Tenants.MemberIDPrefix` + sequence → `MW000123`). Backfill all existing agents. Display the new code wherever agent identity is shown to operators, agents, and members. Do not change enrollment-link generation in this scope.

## Goals

1. Every `oe.Agents` row has a non-null, human-readable `AgentCode`.
2. The code is consistent across creation paths (manual, agency-duplication, future imports).
3. Each tenant has an editable prefix (default falls back to a global default when blank).
4. The code is visible to TenantAdmins, agents themselves, and members in the surfaces enumerated below.
5. Agency-duplication retains all existing functionality; the placeholder `DUP...` codes are retired.

## Non-Goals

- Replacing or deprecating `NPN`. NPN remains a separate optional column for regulatory use.
- Replacing `AgentId` (UNIQUEIDENTIFIER) as the internal primary key. All foreign keys continue to use `AgentId`.
- Changing how enrollment-link templates resolve their displayed code (currently NPN-based — out of scope, see Future Work).
- Exposing `AgentCode` in public URLs. Agents are not enumerated externally; the value lives in lists, profiles, and dashboards behind authentication.
- Public-facing agent profile pages or directory.

## Background

### Current `oe.Agents.AgentCode` state (production)

- Column: `NVARCHAR(100) NULL` — nullable, **no unique index**, no constraints.
- Population: 176 total agents — **172 NULL/empty**, **4 DUP-prefixed**, **0 other**, 130 have an NPN.
- The 4 DUP codes were generated by `agencyAdminProvisioning.service.js:343` using `DUP{base36 timestamp}{4-char base36 random}` (e.g., `DUPMO3EPEUSB31A`).
- No code path branches on `LIKE 'DUP%'`. The DUP codes have no semantic meaning — they were chosen to give the duplication path *some* unique non-null value.

### Current display surfaces for `AgentCode`

- `frontend/src/components/enrollment-wizard/steps/BasicInfoStep.tsx:586` — agent picker dropdown (operator-facing).
- `frontend/src/pages/groups/GroupsAddGroup.tsx:370` — fallback when NPN missing (operator-facing).
- Several enrollment-link template paths conflate NPN with AgentCode (`backend/routes/me/sysadmin/enrollment-link-templates.js:951`, `backend/routes/tenants.js:972`, `frontend/src/services/enrollment-link-templates.service.ts:432,485`). **Not touched in this scope.**

### Existing precedent: Member IDs

- Prefix lives on `oe.Tenants` as `MemberIDPrefix` and `IndividualMemberIDPrefix`.
- Generation handled by stored procedure `oe.GenerateHouseholdMemberID` (`sql-changes/2026-04-29-listbill-individual-prefix.sql`), with global per-prefix sequence and `OED` fallback when prefix is unset.
- Format: `{PREFIX}{SEQUENCE}` (sequence is monotonically increasing per prefix).

## Design

### Format

```
{PREFIX}{6-digit zero-padded sequence}
```

- Example: `MWA000124` (custom tenant prefix), `AW000001` (default fallback prefix).
- Prefix: 1–6 uppercase alphanumeric characters, configured per tenant.
- Sequence: 6 digits, zero-padded, room for ~1 million agents per prefix before width overflow (acceptable; can extend later).
- Default prefix when `oe.Tenants.AgentIDPrefix` is null/empty: **`AW`** (AllAboard Wide, applies to all tenants without a custom prefix).
- Sequence scope: **global per prefix** (matches Member ID pattern). Two tenants sharing a prefix would share a sequence — operationally avoided by giving every tenant a distinct prefix.

### Database changes (`sql-changes/2026-05-07-agent-ids.sql`)

1. Add `oe.Tenants.AgentIDPrefix NVARCHAR(10) NULL`.
2. Create stored procedure `oe.GenerateAgentCode(@TenantId UNIQUEIDENTIFIER) RETURNS NVARCHAR(50)` modeled on `oe.GenerateHouseholdMemberID`:
   - Resolves prefix from `oe.Tenants.AgentIDPrefix`, falling back to `AW`.
   - Atomically picks the next sequence number for that prefix (uses `MAX(CAST(SUBSTRING(...) AS INT))` over `oe.Agents.AgentCode WHERE AgentCode LIKE @Prefix + '%'`, inside a serializable transaction or via a dedicated sequence table — match the existing Member ID approach for consistency).
   - Returns the formatted code.
3. Add a non-unique index `IX_Agents_AgentCode` on `oe.Agents(AgentCode)` to support prefix-LIKE queries.
4. **Backfill script** (idempotent, runs once):
   - Iterate `oe.Agents` ordered by `CreatedDate ASC`, grouped by `TenantId`.
   - For each row whose `AgentCode` is null/empty **or matches `'DUP%'`**, assign the next sequence value for that tenant's prefix.
   - The 4 DUP-coded production rows get fresh sequential codes alongside the 172 nulls.
   - Records the mapping (old → new) in a one-off audit table or log file for support reference.

### Backend changes

#### Service layer

- New `backend/services/agentCode.service.js` exposing `generateAgentCode(tenantId, transactionOrPool)`:
  - Calls the stored procedure.
  - Returns the new code as a string.

#### Write paths (must call `generateAgentCode`)

1. `backend/routes/tenant-admin-agents.js:~1895` (manual TenantAdmin agent creation) — currently leaves AgentCode null. **Change:** generate and insert.
2. `backend/services/agencyAdminProvisioning.service.js:343` (agency duplication) — currently produces `DUP...`. **Change:** replace with call to `generateAgentCode`. **The functional behavior of duplication is unchanged** — only the format of the resulting code differs.
3. Any other agent-insert path discovered during implementation (e.g., bulk import, sysadmin provisioning) — same treatment.

#### Read paths

- Existing routes already select `AgentCode` (`backend/routes/agents.js:383`, `backend/routes/me/tenant-admin/enrollment-link-templates.js:2023`). No change needed — they will start returning the new values automatically.
- Add `AgentCode` to the agent-detail endpoint and to the member's "Your Agent" endpoint if not already present.

#### Tenant prefix management

- Extend the existing tenant-edit endpoint (`backend/routes/tenants.js`) to accept `AgentIDPrefix`.
- Validation: 1–6 uppercase alphanumeric characters, no spaces, no special characters. Editable by SysAdmin only (matches MemberIDPrefix permission model).
- Changing the prefix does **not** retroactively rename existing agents; it only affects future generations. (Same behavior as MemberIDPrefix today.)

### Migration / backfill plan

1. Deploy DB migration: add column + stored procedure + index.
2. Deploy backend with the new write-path wiring (still works pre-backfill — new agents get codes; old agents stay null until backfill runs).
3. Run backfill script (one-shot): assigns codes to the 172 nulls and 4 DUPs. Logs old→new mapping.
4. Deploy frontend with the new display surfaces.
5. Verify in testing first, then promote to prod.

The order is chosen so the new write paths and backfill can complete before any UI surfaces them — avoiding a window where users see blank AgentCode columns.

### Testing

- **Backend Jest:** unit tests for `agentCode.service.js` (prefix fallback, padding, collision behavior). Integration tests for the manual-creation route and the agency-duplication path proving they assign sequential codes.
- **Vitest:** snapshot/render tests for the new TenantAdmin column, agent profile field, member "Your Agent" card.
- **Cypress:** one happy-path spec where a TenantAdmin creates an agent and sees the assigned AgentCode appear in the list.
- Per user feedback memory: minimize test runs during development; run focused tests before PR.

## Edge cases

- **Two agents created concurrently in the same tenant:** stored procedure must be atomic (serializable read of `MAX(...)` + insert, or `SELECT ... FOR UPDATE` equivalent) so two sequential codes don't collide.
- **Tenant has no `AgentIDPrefix` set:** fall back to `AW`. Multiple tenants without a prefix will share the `AW` sequence — acceptable, matches Member ID behavior.
- **Tenant changes their prefix later:** existing agents keep their old code; new agents get the new prefix. (Sequence resets per prefix, so the first agent under the new prefix is `{NEWPREFIX}000001`.)
- **Soft-deleted agents:** sequence does not reuse numbers. Backfill includes inactive/terminated agents so support can reference them.
- **Backfill rerun safety:** script must skip rows that already have a non-DUP, non-empty AgentCode so it's idempotent.
